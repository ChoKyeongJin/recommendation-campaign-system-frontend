00한눈에 보기

타겟팅이란 — “누구에게 보낼지”를 찾는 일

캠페인의 첫 단추는 “누구에게 보낼 것인가”다. 담당자는 SQL을 몰라도 되고, 고객 조건을 평범한 한국어 문장으로 적기만 하면 시스템이 그 문장을 해석해 실제 고객 데이터베이스에서 대상자를 뽑는 조회문(SQL)을 만들고, 그 SQL을 안전하게 실행해 대상 고객 수와 구성까지 돌려준다.

비유로 이해하기 SQL을 잘 아는 데이터 분석가에게 “20대 여성 중에 장바구니에 담아만 두고 안 산 사람 명단 좀 뽑아줘”라고 부탁하는 것과 같습니다. 분석가는 ① 애매한 말을 정확한 조건으로 정리하고 → ② 어느 테이블·컬럼을 봐야 하는지 판단하고 → ③ 조회문을 짜고 → ④ 실수로 데이터를 지우거나 엉뚱한 사람을 넣지 않았는지 스스로 점검한 뒤 → ⑤ 실행해 명단을 가져옵니다. 이 보고서는 그 “분석가”가 머릿속으로 밟는 10단계를 그대로 풀어 설명합니다.

화면에서 이 과정은 4단계 마법사 중 ①프롬프트 입력 → ②타겟팅 결과 구간에 해당한다. 담당자 눈에는 “문장 입력 → 결과”로 순식간에 보이지만, 그 사이에서 백엔드는 아래 10개의 처리 단계를 순서대로 실행한다.

02핵심 설계 원칙

왜 이렇게 설계했나 — “결정론 우선, AI는 보조”

타겟팅은 고객 명단을 다루는 일이라 “그럴듯한 오답”이 곧 잘못된 발송이 된다. 그래서 이 시스템은 세 가지 원칙 위에 서 있다.

원칙 1 · 결정론(규칙)을 먼저, AI는 “말귀” 보조만

조건 해석의 뼈대는 정규식·룰 로직이 잡는다. AI(OpenAI gpt-4o-mini)는 사람 말투를 표준 용어로 바꾸거나(재작성), 규칙이 놓친 상품명을 건져 올리는 등 “유연성(재현율)”만 담당하고, 그 결과는 다시 규칙으로 검증한다. 실제로 AI가 계획을 세워도 그 뒤에 규칙 추출기가 한 번 더 덧씌워 브랜드 코드 해석·구매이력·생일·가입일 같은 핵심 조건을 결정론적으로 확정한다.

원칙 2 · API 키가 없어도 돌아간다 (Graceful Degradation)

모든 AI 단계는 OPENAI_API_KEY가 있는지 먼저 확인하고, 없으면 조용히 규칙 기반 경로로 대체한다. 즉 AI가 죽거나 키가 빠져도 타겟팅은 완전한 결정론 파이프라인으로 계속 동작한다. AI는 품질을 높이는 옵션이지 필수 부품이 아니다.

게이트 규칙 문장 재작성(STEP 1)과 상품 추출(STEP 4)은 API 키 유무만으로 켜진다(전처리라서). 반면 AI 질의계획(STEP 3)·AI SQL 생성(STEP 8)은 키가 있고 동시에 파서 모드가 auto/llm일 때만 쓴다. 운영 기본값은 안전하게 rules 경로다.

원칙 3 · 지어내지 않는다 (검증 + 부분추출 고지)

AI가 뽑은 상품명은 원문에 실제로 등장할 때만 인정하고(없는 상품 차단), 생성된 SQL은 4중 안전 검증을 전부 통과해야 한다. 조건을 데이터로 매핑하지 못하면 몰래 빼지 않고 “이 조건은 반영하지 못했습니다”라고 미지원·제외 조건으로 고지한다. “대충 맞는 명단”보다 “무엇이 반영되고 무엇이 빠졌는지 아는 명단”을 지향한다.

일반 “LLM에게 SQL 맡기기”

사용자 문장 → LLM이 알아서 SQL 생성 → 바로 실행

빠르지만 환각(없는 컬럼·조건)·비재현성·안전성 위험. 왜 그 명단이 나왔는지 설명 불가.

이 시스템 (결정론 우선 + 그래프 근거)

문장 → 규칙 파싱(+AI 보조) → 그래프에서 실제 스키마 근거 검색 → 검증된 템플릿 SQL → 4중 검증 → 실행

재현 가능·안전·설명 가능. 조건별 근거와 신뢰도까지 남긴다.

03STEP 1 · 프롬프트 재작성

담당자의 말투를 표준 타겟 용어로 정리

파싱을 시작하기 전에, 구어체·오타·모호한 표현을 표준화된 타겟팅 문장으로 다시 쓴다. 원문은 그대로 보존하고, 이 재작성본이 이후 모든 해석의 기준이 된다.

하는 일 (쉽게)

사람마다 같은 뜻을 다르게 쓴다. “2030 여성인데 장바구니만 담고 안 사는” → “20~30대 여성 장바구니 이탈 고객”처럼, 조건을 더하거나 빼지 않으면서 표현만 표준 용어로 통일한다. 뒤에 오는 규칙 파서는 “정해진 표준 용어”만 정확히 잡아내기 때문에, 이 정리 단계가 조건 인식률을 실제로 끌어올린다.

04STEP 2 · 타겟 / 채널 분리

“누구에게”와 “무엇을 어떻게”를 갈라낸다

한 문장에는 대상 조건(누구를 뽑을지)과 발송 조건(어떤 채널로 무엇을 보낼지)이 섞여 있다. 타겟팅 SQL에는 대상 조건만 필요하므로 둘을 분리한다.

하는 일 (쉽게)

“20대 여성에게 신상 화장품을 문자로 보내는 캠페인”에서, SQL로 명단을 뽑을 때 필요한 건 “20대 여성”뿐이다. “문자(채널)”나 “신상 화장품(보낼 것)”이 대상 조건 파싱에 섞이면, 엉뚱하게 채널 단어가 검색을 오염시키거나 상품이 타겟 조건으로 잘못 잡힌다. 그래서 먼저 갈라놓는다.

동작 조건타겟팅 조회(retrieval_scope="targeting", 운영 기본)에서는 타겟 절만 파싱해 SQL로 만든다

05STEP 3 · 질의 계획 수립

문장을 “구조화된 조건표”로 바꾼다 — 핵심 단계

정리된 타겟 문장을, 기계가 다룰 수 있는 조건표(Query Plan)로 변환한다. 이 단계가 타겟팅의 심장이며, 규칙 파서가 먼저 전부 돌고 AI는 그 위에 얹히는 하이브리드 구조다.

Query Plan에 담기는 것

  • intent — 무엇을 하려는가 (고객 세그먼트 찾기 / 캠페인 추천 등)
  • target_user — 성별, 연령(“20대” → 20~29), 라이프사이클(신규·휴면 등), 관심사, 행동, 구매이력(구매 상품), 미활동 기간, 생일, 가입 시점
  • exclude — 제외 조건
  • campaign_constraints — 캠페인 목적·오퍼·판매 상품(발송 맥락)
  • dimension_filters / set_expressions / computed_metrics — 값 해석·집합식·계산 지표(STEP 5·6에서 상세)

규칙 우선 → AI 보조 → 규칙 재적용 (3중 구조)

  1. 규칙 파서가 먼저 전부 실행 (_build_rule_query_plan) 정규화 사전으로 동의어를 표준값으로 바꾸고, 연령·구매이력·미활동·생일·가입·지역·지표 등을 정규식으로 추출한다. 이 결과만으로도 완결적인 계획이 나온다.
  2. (선택) AI 파서가 보강 (_try_llm_query_plan, auto/llm 모드에서만) OpenAI가 규칙 계획을 fallback으로 참고해 더 정교한 계획을 만들고, 화이트리스트 검증으로 허용된 필드만 남긴다.
  3. AI 결과 위에 규칙 추출기를 다시 덧씌움 브랜드→코드 해석·구매이력·생일·가입 창(window) 등 핵심 조건은 파서 모드와 무관하게 항상 결정론으로 확정 — AI 오분류를 막는 안전장치.
다중 표현 병합 (옵션) 한 문장을 AI로 여러 방식으로 바꿔 말한 뒤 각각을 규칙으로 파싱해 조건을 합집합으로 병합하는 옵션도 있다. 한 가지 표현에서 놓친 조건을 다른 표현에서 건져, 조건 커버리지를 높인다.
이 예시에서는 · STEP 3
들어올 때타겟 절 서울 거주 20대 여성 중 기저귀 구매 고객
조건표목적 = 캠페인 발송
성별 = 여성 · 연령 = 20~29세 · 지역 = 서울
구매 상품 = 기저귀

06STEP 4 · 상품·구매이력 추출

“무엇을 산/살 사람인가” — 정규식 우선 + 검증된 AI 보조

문장에서 상품·카테고리를 뽑아 구매이력 조건(과거에 그 상품을 산 고객)과 판매 상품(캠페인이 팔려는 상품)으로 나눈다. 재작성이 표현을 다양화하므로, 고정 정규식만으로는 놓치는 경우를 AI가 메운다.

왜 하이브리드인가

STEP 1 재작성이 “기저귀를 산 고객”을 “기저귀 구매 고객”처럼 명사형으로 바꾸면, 동사 위주 정규식이 조용히 놓쳐 “타겟 조건을 못 찾아 SQL 실패”가 반복됐다. 그래서 정규식이 놓치고 구매/판매 신호가 있을 때만 AI를 호출해 상품명을 건진다 — 단, 원문에 실제로 등장하는 단어만 인정해 없는 상품을 지어내지 못하게 한다.

역할 분담담당
재현율(표현 유연성)AI(OpenAI) — 정규식이 놓친 명사형·변형 표현을 잡음
정밀도(환각 차단)규칙 _validated_object() — 추출 단어가 원문에 있어야만 채택
비용/지연 최소화정규식이 이미 뽑았거나 구매·판매 신호가 없으면 AI 호출 안 함
게이트OPENAI_API_KEY 유무로 동작(파서 모드 무관). TARGET_OBJECT_LLM_FALLBACK=false로 끔

회귀 방지: 새 실패 케이스는 테스트 코퍼스(tests/test_target_object_extraction.py)에 “문장 → 기대 추출” 한 줄로 추가해 재발을 막는다.

이 예시에서는 · STEP 4

안전장치‘기저귀’는 원문에 실제로 있는 단어라 채택 (원문에 없는 상품이면 지어낸 것으로 보고 거부)

07STEP 5 · 값 해석 (브랜드명 → 코드)

사람이 쓰는 이름을 DB가 아는 코드값으로

데이터베이스는 “포멜카멜리”라는 이름이 아니라 코드값 'A'로 브랜드를 저장한다. 프롬프트의 디멘션 값(브랜드·지역 등)을 실제 코드로 실시간 변환하는 단계 — 100% 규칙(AI 미사용)이다.

변환 경로 (예: 상품브랜드 “포멜카멜리”)

  1. 디멘션 식별프롬프트에 “상품브랜드” 같은 디멘션 라벨/동의어가 있으면 해당 디멘션을 찾는다.
  2. DS_SQL 실행으로 값 목록 조회그 디멘션의 조회 SQL을 실제 DB에 돌려 (코드, 이름) 목록을 받는다. 규칙: 첫 컬럼=코드, 둘째=이름. 결과는 캐시.
  3. 이름 → 코드 매칭“포멜카멜리” → 'A'. 최종 조건은 BRAND_ID IN ('A').
항목내용
함수_apply_dimension_filters() → _resolve_dimension_values_cached() (LRU 캐시)
카탈로그dimension_catalog.sample.json — DS 디멘션 정의만 담고, 값은 저장하지 않고 런타임에 동적 조회(값이 많을 수 있어서)
라벨 없는 값지역명처럼 라벨이 없는 값은 member_value_index.json으로 매칭. “경기”가 “경기침체” 안에 걸리지 않도록 토큰 경계 검사 적용
AI 사용없음 — 값 해석은 실제 SQL 조회이지 AI 추론이 아니다
이 예시에서는 · STEP 5
값지역명 서울
해석회원 지역 컬럼(SIDO)의 값으로 인식 → 조건 SIDO = '서울'
브랜드라면가령 포멜카멜리 였다면 → 사전 조회 SQL 실행 → 코드 'A' → BRAND_ID IN ('A') (이 단계의 본령)

08STEP 6 · 집합식 파싱

“A 제외”, “그중 B만” 같은 집합 연산 해석

여러 세그먼트를 합치고(합집합), 겹치고(교집합), 빼는(차집합) 한국어 집합 표현을 연산 구조로 파싱한다. 순수 규칙(정규식 + 연산자 우선순위)으로 처리한다.

표현연산
또는 / 합집합합집합(union)
그리고 / 교집합교집합(intersection)
제외 / 빼고 / 차집합차집합(difference)
대상으로 하되 X만 포함 / 그중 / 중에서자연어 정제(refinement)
과잉 인식 방지 (실사례) 단순한 “대상으로”는 집합 연산으로 보지 않는다. 과거 재작성이 “20대 여성 고객을 대상으로 …”처럼 바꾸자 파서가 이를 20대 ∩ 여성 교집합으로 오인해 연령·성별이 통째로 사라지고 명단이 비는 버그가 있었다. “포함/남기고/제외/그중” 같은 진짜 집합 표지가 있을 때만 집합식으로 성립하도록 고쳤다. (회귀 테스트로 고정)
이 예시에서는 · STEP 6
판정“…여성 중 기저귀 산” 의 “중” 은 자연스러운 정제 표현일 뿐, 집합 표지가 아님
결과집합 연산 없음 — 건너뜀 (연령·성별이 통째로 사라지는 오인을 막는다)
집합이라면가령 …여성 중 VIP는 빼고 였다면 → 차집합(제외) 연산으로 처리

09STEP 7 · 지식그래프 검색 (GraphRAG)

SQL을 짜는 데 필요한 “근거”를 찾아온다

조건을 실제 테이블·컬럼으로 옮기려면 스키마 지식이 필요하다. 이를 위해 793개 노드의 지식 그래프에서 관련 스키마·조인·예시를 벡터 + 키워드 검색 → 그래프 확장으로 찾는다.

검색 3단계

10STEP 8 · 타겟팅 SQL 생성

검증된 템플릿 우선, 안 맞으면 AI가 근거 위에서 생성

지금까지 모은 조건표와 그래프 근거로 최종 CRM SQL을 조립한다. 결정론 템플릿이 1순위, 템플릿으로 표현 못 하는 형태만 AI가 2순위 폴백으로 생성한다.

1순위 · 결정론 템플릿 (의도별 라우팅)

build_sql_template_candidate()가 조건에 맞는 템플릿을 순서대로 시도한다: 장바구니-브랜드 → 장바구니 재구매(담고 안 산) → 구매이력 → 주문횟수(첫구매·재구매·미구매) → 지표 랭킹(매출 상위 N) → 일반 회원속성. 모두 compile_member_target_conditions()를 공유해 조건을 회원 허브 테이블(CRM_MB_BASEINFO) 조건으로 변환한다.

예 · “카트에 담고 안 산” 계열 템플릿의 골격:

SELECT DISTINCT B.MEMBER_NO AS CUST_ID, ... , 'cart_abandoner' AS target_segment
FROM   ODS_MALL_OMS_CART A
       INNER JOIN CRM_MB_BASEINFO B ON A.CART_ID = B.MEMBER_ID
WHERE  A.KEEP_YN = 'Y'                                  -- 담아둔 채 미결제
   AND B.GENDER_CD = 'GENDER_CD.FEMALE'                 -- 여성
   AND B.AGE >= 20 AND B.AGE <= 29                      -- 20대
   AND B.MEMBER_STATE_CD = 'MEMBER_STATE_CD.NORMAL'     -- 정상 회원 기본 게이트

2순위 · AI SQL 폴백 (템플릿이 없을 때만)

어떤 템플릿에도 맞지 않고 파서 모드가 auto/llm일 때만, OpenAI가 그래프 검색 근거(상위 컨텍스트 + 허용 테이블 목록) 안에서만 SQL을 만든다. 규칙: MSSQL SELECT 전용, 첫 컬럼 B.MEMBER_NO AS CUST_ID, 정상 회원 게이트 기본, 언급되지 않은 조건 추가 금지. 이 결과에는 llm_generated 꼬리표가 붙는다.

공통 · 부분추출 고지 데이터로 매핑하지 못한 조건은 몰래 빼지 않고 미지원(unsupported)·제외(dropped) 조건으로 표시해 담당자에게 알린다. 또 confidence.py가 조건별 근거와 함께 0~100 신뢰도 점수를 붙인다.
이 예시에서는 · STEP 8
템플릿 선택상품 구매 이력 결정론 템플릿 (구매 상품 조건이 있으므로)
만든 조회주문상세 ⨝ 상품 ⨝ 회원 을 이어서 —
상품명에 ‘기저귀’ 포함 · 성별 여성 · 20~29세 · 지역 서울 · 정상 회원
고지데이터로 못 옮긴 조건이 있으면 몰래 빼지 않고 ‘미반영’으로 표시 + 신뢰도 점수 부여

11STEP 9 · SQL 안전 검증

실행 전, 4중 관문을 통과해야만 한다

템플릿이든 AI든 모든 SQL 후보는 동일한 검증 관문을 통과해야 실행된다. 하나라도 걸리면 그 후보는 탈락한다.

  1. 안전성 검증 (validate_sql · sql_guard.py) 읽기(SELECT)만 허용 — INSERT·UPDATE·DELETE·DROP 등 데이터 변경 명령 전면 차단, 문장 끼워넣기 차단, 스키마에 등록된 허용 테이블만 접근, 이름·전화 등 민감 컬럼은 마스킹 SQL로 대체.
  2. 조건 커버리지 검증 (validate_sql_condition_coverage) 사용자가 요구한 조건(연령·성별·상품 등)이 SQL에 실제로 모두 반영됐는지 확인. 빠졌으면 탈락.
  3. 의도 범위 검증 (validate_sql_intent_scope) SQL이 문장의 의도(세그먼트 찾기/추천 등)와 맞는 범위인지 확인.
  4. 미언급 조건 차단 (validate_unmentioned_sql_conditions) 사용자가 말하지 않은 조건을 SQL이 임의로 추가했는지 검사해 차단. “시키지 않은 필터링”을 막는다.

네 관문을 모두 통과한 후보만 “사용 가능(eligible)”으로 인정된다. 대량 조회 방지를 위해 조회 건수 상한도 함께 관리한다.

이 예시에서는 · STEP 9
① 안전성읽기(SELECT)만 · 허용된 주문/상품/회원 테이블만 접근 → OK
② 커버리지성별·연령·지역·기저귀 가 SQL 에 모두 반영됐나 → OK
③ 의도‘발송 대상 뽑기’ 범위와 일치 → OK
④ 미언급말하지 않은 조건을 임의로 넣지 않았나 → OK
결과4중 관문 통과 → 실행 가능

13전 과정 워크스루

“서울 사는 20대 여성 중에 기저귀 산 사람한테 쿠폰 문자 보내줘” 따라가기

앞 장들의 “이 예시에서는” 상자를 한 줄로 이어, 이 한 문장이 10단계를 지나 대상 명단이 되는 전 과정을 다시 본다.

  1. 재작성 (STEP 1)구어체를 “서울 거주 20대 여성 중 기저귀 구매 고객에게 쿠폰 문자 발송”으로 정리, 원문은 보존.
  2. 타겟/채널 분리 (STEP 2)대상 = 서울 거주 20대 여성 중 기저귀 구매 고객, 발송 = 쿠폰 문자 — 대상 절만 파싱.
  3. 질의 계획 (STEP 3)성별 여성 · 연령 20~29 · 지역 서울 · 구매 상품 기저귀 · 목적 = 캠페인 발송, 으로 구조화.
  4. 상품 추출 (STEP 4)“산” = 구매 신호 → 구매 상품 기저귀(원문에 있는 단어라 채택).
  5. 값 해석 (STEP 5)“서울” → 회원 지역 컬럼 값 SIDO = '서울'로 해석.
  6. 집합식 (STEP 6)“중”은 정제 표현이라 집합 연산 없음 — 연령·성별을 정상 조건으로 유지.
  7. 그래프 검색 (STEP 7)주문상세·상품·회원 테이블과 “상품 구매 이력” SQL 예시 근거를 확보.
  8. SQL 생성 (STEP 8)상품 구매 이력 템플릿 → 주문상세 ⨝ 상품 ⨝ 회원, 상품 ‘기저귀’ + 여성 + 20~29 + 서울 + 정상회원.
  9. 안전 검증 (STEP 9)읽기 전용·허용 테이블·조건 커버리지·미언급 조건 검사 통과.
  10. 실행·결과 (STEP 10)CRMDW에 실행 → 대상 고객 수 + 세그먼트 구성 반환. 화면 “타겟팅 결과”에 표시.

docs/data JSON은 언제·왜 들어가나 — 파이프라인의 “연료”

앞의 03~12장이 “어떻게 처리하는가”였다면, 이 장은 그 처리에 “어떤 사전·지식 파일이 연료로 들어가는가”다. docs/data 폴더의 JSON은 코드가 아니라 데이터라, 이 파일만 고치면 재배포 없이 타겟팅 동작(인식하는 표현·값 매핑·허용 테이블 등)을 바꿀 수 있다. 각 파일이 어느 STEP에서 · 어떤 함수로 · 왜 들어가는지, 그리고 비었거나 매핑이 없으면 무슨 일이 나는지까지 정리한다. (실물은 화면 설정 → 참조 파일에서 읽기 전용으로 열람 가능.)

먼저 · 런타임 vs 빌드타임 대부분의 JSON은 질의마다 런타임에 직접 로드된다(파일 → 사전). 예외는 rag_knowledge_base.json 하나로, 이건 오프라인 빌드 산출물이다 — build_rag_knowledge.py가 아래 원천 파일들(스키마·정규화·정책·지표·SQL예시·디멘션)을 미리 묶어 지식그래프로 굳힌 것. 그래서 STEP 7 검색이 실제로 뒤지는 실체는 이 한 파일이고, 원천 파일들은 “빌드 재료”이자 “런타임 사전”의 두 역할을 겸한다.

A · “말귀”를 표준 조건으로 — STEP 2·3·4 사전

targeting_lexicon.json — 타겟팅 어휘 단일 출처

시스템이 인식하는 모든 타겟팅 표현을 한 곳에 모은 사전. 새 말투·새 판매 동사 등은 코드가 아니라 여기에 추가한다.

  • 들어가는 시점 — STEP 2(대상 방향표지 에게/한테/대상으로로 타겟·채널 절 분리), STEP 3(의도·목표 분류: 추천 캠페인 vs 세그먼트 조회), STEP 4(구매/판매 신호어 감지). 로드: _lexicon_terms()
  • 주요 키 — audience_direction_markers, channel_signal_words, intent_recommend_campaign, objective_rules, sell_outreach_verbs, purchase_history_signals, cart_terms, repurchase_terms …
  • 못 찾으면 — 사전에 없는 새 표현은 그 단계가 통째로 인식하지 못한다. 예: 판매 신호어에 없는 동사면 STEP 4 상품추출의 AI 폴백조차 켜지지 않는다. → 새 표현은 여기 한 줄 추가가 정답.

normalization_rules.sample.json — 표준화 사전

사람이 쓰는 다양한 표면형을 표준값(canonical)으로 접어준다. 예: 여자 / 여성 / 여자 고객 → female.

  • 들어가는 시점 — STEP 3(성별·등급·행동 등 조건을 canonical로 확정), STEP 6(집합식 피연산자 용어 카탈로그 load_set_term_catalog()로 재사용), + 신뢰도 근거(matched_terms)
  • 구성 — 규칙별 rule_id · canonical · ko_label · synonyms(활용·조사 변형 포함)
  • 못 찾으면 — 동의어 미등록 표현은 조건으로 안 잡히거나(누락), 집합식에서 unknown_operand가 되어 SQL 대신 되물음(clarification)이 나간다.

business_policies.sample.json — 비즈니스 정책

  • 역할 — 캠페인 공통 정책(임계값·기본 제약)을 계획에 자동으로 얹는다. 예: 발송 대상 기본 게이트(탈퇴/휴면 제외).
  • 들어가는 시점 — STEP 3 _apply_policy_constraints()로 Query Plan에 주입. 동시에 STEP 7 지식그래프의 정책 노드 원천이기도 하다.

metric_lexicon.sample.json · member_metrics.json — 지표 어휘

  • metric_lexicon.sample.json — 공식 기반 계산지표(예: 객단가 = 매출/주문수)를 문장에서 인식. STEP 3 parse_computed_metrics_from_query()(formula_engine.py).
  • member_metrics.json — 회원 지표 레지스트리. “구매액이 높은 고객/지역” 같은 표현을 지표 동의어로 동적 인식(STEP 3)하고, 랭킹형 타겟 SQL(STEP 8)의 근거가 된다.

B · 값을 코드로 — STEP 5 값 해석

dimension_catalog.sample.json — 디멘션 카탈로그

  • 역할 — “포멜카멜리” 같은 이름을 실제 코드값('A')으로 바꾸는 규약. 디멘션별 synonyms · target_column · ds_sql · connection.
  • 들어가는 시점 — STEP 5 _apply_dimension_filters(). 라벨/동의어가 프롬프트에 있으면 런타임에 DS_SQL을 실DB에 실행해 (코드,이름) 쌍을 얻는다.
  • 못 찾으면 — 라벨이 없으면 이 경로 자체를 건너뛰고, DS_SQL 실행 실패는 조용히 빈 결과로 흐른다(진단 사각지대). 값만 있을 땐 아래 역인덱스가 보완.

member_value_index.json — 회원 값 역인덱스

  • 역할 — 라벨 없이 값만 언급될 때(예: 라벨 없이 브랜드명만) 그 값이 어느 컬럼에 속하는지 찾아준다.
  • 들어가는 시점 — STEP 5 _apply_member_value_filters(). 같은 이름이 여러 컬럼에 있으면(모호) 안전하게 제외. 생성: build_member_value_index.py.

C · SQL을 만들고 지키는 기준 — STEP 8·9

member_target_filters.json — 조건→컬럼 매핑의 핵심 사전

  • 들어가는 시점 — STEP 8 compile_member_target_conditions()가 이 매핑으로 결정론 SQL을 조립. + 신뢰도 채점의 근거.

  • 못 찾으면(가장 중요) — 매핑이 없는 조건은 ‘실DB 미지원’으로 분류돼 SQL에서 빠지고 “이 조건은 반영 못 함”으로 고지된다. → 커버리지를 넓히는 1순위 손잡이가 바로 이 파일.

rag_knowledge_base.json — 지식그래프 본체

JSON은 아니지만 함께 쓰는 원천 같은 폴더의 비-JSON 3종도 지식 빌드에 들어간다 — sql_examples.sample.sql(검색 근거·LLM SQL 폴백 참고), table_relationships.md(조인 경로 문서), metadata_ddl.sql(프롬프트·정책 저장용 메타 DB 스키마).
