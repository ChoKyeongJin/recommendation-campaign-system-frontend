00한눈에 보기

이 시스템이 하는 일 — 쉽게

담당자가 “어떤 고객에게, 어떤 캠페인을 보내고 싶다”를 평범한 문장으로 적으면, 시스템이 알아서 ① 보낼 대상 고객을 찾는 조회문(SQL)을 만들고 ② 문자 메시지 문구 3개를 써주고 ③ 어떤 문구가 가장 많이 눌릴지(클릭률)를 예측해 추천해 줍니다.

비유로 이해하기 똑똑한 마케팅 신입사원이 옆에 있다고 생각하면 됩니다. 말로 지시하면 — 대상 명단을 뽑고(타겟팅), 보낼 문자 초안 3개를 쓰고(메시지 생성), “제 생각엔 B안이 제일 반응 좋을 것 같아요”라고 근거까지 붙여 추천(클릭률 예측)합니다. 다만 지어내지 않도록 여러 안전장치를 걸어 뒀습니다.

이 문서는 그 신입사원이 속으로 어떤 순서로 일하는지를 단계별로 풀어 설명합니다. 화면에서는 프롬프트 입력 → 타겟팅 결과 → 메시지 추천 → 클릭 분석 4단계로 보입니다.

02지식베이스 구축

지식베이스(RAG) 구축

DDL을 출발점으로 스키마 카탈로그와 정규화 사전을 만들고, 이를 합쳐 RAG 지식베이스를 구성한 뒤 Qdrant에 색인한다.

구축 프로세스

  1. DDL에서 스키마 카탈로그 생성 CREATE_TABLE, CREATE_INDEX, PK/FK/check 제약을 DDL에서 자동 추출한다.
  2. 스키마 설명과 중요 컬럼 메모 보강 테이블 설명과 중요 컬럼 메모가 좋아질수록 자연어 질문이 올바른 테이블·컬럼으로 이어질 확률이 높아진다.
  3. 정규화 사전 검증 한국어 표현, 영어 표현, 부정 동의어를 canonical 값으로 바꾼다.
  4. RAG 지식베이스 생성 RAG에 넣을 데이터를 준비하는 단계.
  5. 입력 데이터 사전 검증 임베딩 생성이나 Qdrant 쓰기 없이 노드 개수, 타입 분포, 샘플 ID를 확인한다.
  6. Qdrant 컬렉션 초기화와 색인 실제로 임베딩과 Qdrant 색인이 이루어진다.
  7. Qdrant 컬렉션 상태 점검 Qdrant에 저장은 했는데 정말 제대로 저장됐는지 확인한다.

04질의 프로세스

질의(Query) 프로세스

질의 전 그래프 상태를 점검하고, GraphRAG로 질문을 검색·확장·조립한 뒤, 각 단계가 무엇을 만들어냈는지 확인한다.

1) 그래프 통계 확인

질의를 하기 전에 그래프가 정상적으로 만들어졌는지 확인하는 단계. 노드(Node) 개수, 관계(Edge) 개수, 이상 여부를 점검한다.

2) GraphRAG 질의 실행

  • Query Plan — 무엇을 검색할지 결정한다.
  • 벡터 검색 — Qdrant에서 의미가 비슷한 문서를 검색한다.
  • 키워드 검색 — 키워드 기반 검색 (BM25나 Full-text Search 같은 문자열 기반 검색).
  • 그래프 확장 — 관련 노드까지 그래프를 따라 확장한다.
  • Context 조립 — 검색 결과를 하나의 컨텍스트로 합친다.
  • SQL 템플릿 검증 — 사용 가능한 SQL 예시인지 확인한다.

3) 결과 확인 포인트

  • 사용자 질문에서 어떤 조건이 추출되었는지 확인한다.
  • 각 단계가 몇 개의 hit / candidate / context를 만들었는지 확인한다.
용어의미
hit검색에서 찾아낸 결과.
candidate검색 결과(hit)에서 그래프를 확장하면 더 많은 관련 노드가 생기는데, 그 후보.
contextLLM에 실제로 넘기는 자료.

타겟팅 SQL 생성 흐름

의미층 → 스키마 허브 → 인스턴스로 이어지는 그래프 연결을 따라, 자연어 질문에서 추출한 조건을 실제 테이블·컬럼에 매핑하고 최종 타겟팅 SQL을 구성한다.

참고 · BM25 검색 결과의 관련성 점수를 계산하는 알고리즘. ① 검색어가 얼마나 많이 등장하는지(Term Frequency), ② 그 단어가 전체 문서에서 얼마나 희귀한지(IDF), ③ 문서가 너무 길지는 않은지 등을 함께 고려해 점수를 계산한다.

05SQL 안전 검증

SQL 안전 검증 프로세스

AI가 만든 조회문(SQL)을 그냥 실행하지 않는다. sql_guard.py가 문지기처럼 한 번 더 검사해서, 데이터를 읽기만 하고 절대 바꾸거나 지우지 못하도록 막는다.

  • 읽기(조회)만 허용 — SELECT 또는 WITH로 시작하는 조회문만 통과시킨다.
  • 데이터를 바꾸는 명령은 전면 차단 — INSERT·UPDATE·DELETE·DROP·ALTER·CREATE·TRUNCATE·MERGE 등 8종 키워드가 있으면 거부한다.
  • 한 번에 한 문장만 — 여러 SQL을 세미콜론으로 이어 붙이는 방식(문장 끼워넣기)을 막는다.
  • 허용된 테이블만 접근 — 스키마에 등록된 테이블만 읽을 수 있다. (단 WITH로 임시 정의한 이름은 예외로 인정)
  • 대량 조회 방지 — LIMIT이 없으면 기본값 100건을 자동으로 붙이고, 조건(WHERE) 없는 전체 조회는 제한한다.
  • 개인정보 보호 — 이름·이메일·전화·주소·생년월일·비밀번호 등 민감 컬럼이 있으면 가려진(마스킹) SQL을 대신 제안한다.
  • 최종 SQL도 반드시 이 검사를 통과해야 사용자에게 반환된다.

검증 결과는 is_valid(통과 여부), safe_sql, masked_sql(가림 처리본), tables, sensitive_columns, issues로 돌려준다.

06선택 근거

GraphRAG를 쓰는 이유

일반 RAG

사용자 질문 → 비슷한 문서 검색 → 문서 내용만 사용 → 기본 추천

GraphRAG

사용자 질문 → 비슷한 문서 검색 → 문서 + 고객 + 상품 + 캠페인 관계까지 탐색 → 개인 맞춤형 추천

문서만 검색하면 캠페인 정보는 찾을 수 있지만, 다음과 같은 관계를 알기 어렵다.

  • 고객이 어떤 상품을 구매했는지
  • 어떤 고객군에 속하는지
  • 어떤 캠페인에 반응했는지
  • 비슷한 고객은 어떤 캠페인에 참여했는지

08메시지 생성

메시지 생성

타겟 고객과 캠페인 문맥을 근거로 여러 메시지 후보(Variant)를 LLM으로 생성하고, 정책·품질·길이를 검증해 채널에 적합한 메시지만 반환한다.

생성 흐름

  1. Message Context메시지 생성을 위한 상황·설정 정보(목적, 타겟, 채널, 추가 정보).
  2. Message PromptContext 데이터를 넣어 프롬프트를 생성한다.
  3. OpenAI Variant GenerationOpenAI(Chat Completions)를 통해 여러 후보 메시지를 생성한다.
  4. Validation금칙어·정책, 길이 제한, 채널 가이드, 중복·유사도, 이모지·특수문자 규칙을 검증·필터링한다.

사용자별 클릭 확률 예측

각 사용자에 대해 A·B·C 메시지의 예상 클릭 확률을 계산한다. 지금 운영 기본값은 “사람이 만든 규칙(heuristic)” 방식이다 (모델 버전 heuristic-ctr-v1). 기본 확률 2.5%에서 출발해, 궁합이 맞는 조건마다 점수를 더하거나 빼서 최종 확률을 만든다 — 마치 체크리스트로 점수를 매기는 것과 같다.

업데이트 · ML 모델은 이미 만들어져 있다 보고서 초안에는 “ML은 나중에 도입 검토”라고 적혀 있었지만, 현재는 LightGBM 기반 클릭률 예측 모델의 학습 파이프라인과 데모 모델(ctr-lgbm-local-demo, 24개 특성·학습 436건)까지 만들어져 있다. 다만 데이터가 아직 적어 운영 기본값은 안전하게 규칙 기반으로 두고, ML 예측이 실패하면 자동으로 규칙 기반으로 되돌아가도록(fallback) 설정했다. 데이터가 충분히 쌓이면 기본값을 ML로 전환할 수 있다. → 아래 “클릭률 예측 모델” 참고

가산·감산 규칙 (기본 확률 2.5%에서 출발, 최종 0.1%~25% 사이로 보정)

  • 사용자가 선호하는 채널이면 +1.2%p
  • 사용자 관심사와 캠페인 카테고리가 맞으면 +1.0%p
  • 가격 민감도가 높고 메시지에 할인·가격 혜택이 있으면 +1.2%p
  • 긴급성 문구와 장바구니·딜 행동이 맞으면 +0.9%p
  • 개인화 문구와 고객 단계(lifecycle)가 맞으면 +0.6%p
  • 메시지 길이가 적당하면 +0.4%p, 너무 길면 −0.4%p
  • 기준(control) 시안이면 소폭 +0.1%p
  • 점수가 같을 때를 대비해 아주 작은 흔들림(최대 ±1%p)을 더해 순위를 가른다

클릭률 예측 모델(LightGBM)과 실제 성과 분석

앞의 “규칙 기반” 방식은 데이터가 적을 때 쓰는 임시 방편이다. 데이터가 쌓이면 진짜 머신러닝 모델로 더 정확하게 예측할 수 있도록, LightGBM 기반 학습 파이프라인을 이미 만들어 뒀다.

두 가지 예측 방식

현재 데모 모델은 학습 436건 규모의 로컬 실습용이다. 운영에서 바로 쓰기엔 데이터가 부족해, 기본값은 규칙 기반으로 두고 ML 예측이 실패하면 자동으로 규칙 기반으로 되돌린다(fallback). 관련 파일은 training/ 폴더에 있다.

모델이 참고하는 정보(특성) 예시

  • 시안 코드(A/B/C), 채널, 캠페인 목적·카테고리
  • 고객의 연령대·성별·지역·고객 단계(lifecycle)·가격 민감도·예상 가치(LTV)
  • 메시지 길이, 발송 시각·요일·주말 여부
  • 선호 채널 일치 여부, 관심 카테고리 일치 여부, 과거 추천 이력 여부

모델 품질 지표 — 무엇으로 “좋다”를 판단하나

모델을 “느낌”이 아니라 숫자로 평가한다. 데이터를 시간 순서대로 학습 70% · 검증 15% · 테스트 15%로 나눠(temporal split) 미래 데이터가 과거 학습에 새는 것(leakage)을 막고, 클릭이 드문 불균형은 scale_pos_weight로 보정하며, 과적합은 조기 종료(early stopping)로 억제한다.

지표무엇을 보나좋은 방향
ROC-AUC클릭할 사람을 안 할 사람보다 위로 잘 세우는가(순위 능력)1에 가까울수록 좋음 · 0.5=찍기
PR-AUC클릭이 드문 상황에서의 정밀도·재현율 균형높을수록 좋음
LogLoss예측 확률이 실제 정답과 얼마나 어긋나는가낮을수록 좋음
Brier확률값 자체의 보정(calibration) 오차낮을수록 좋음
Lift@10%점수 상위 10%만 골랐을 때 전체 평균 대비 클릭률 배수1보다 클수록 좋음

현재 데모 모델(ctr-lgbm-local-demo) 실제 측정값

학습 436건 · 검증 94건 · 테스트 94건 규모의 로컬 실습 모델에서 파이프라인이 실제로 계산한 값이다.

구간ROC-AUCLogLossBrierLift@10%
검증(validation)0.610.390.114.7×
테스트(test)0.310.490.162.9×
정직하게 · 왜 아직 ML을 운영 기본값으로 쓰지 않나 데모 학습 데이터가 436건으로 매우 적어, 테스트 구간 ROC-AUC가 0.31 — 사실상 찍기(0.5)보다 낮게 나온다. 즉 지금 이 모델은 운영에 쓸 수준이 아니며, 그래서 운영 기본값을 규칙 기반으로 유지하는 것이 옳은 선택이다. 중요한 건 학습·평가 파이프라인과 지표 체계가 이미 완성돼 있다는 점이다. 실측 데이터가 쌓여 이 지표들(특히 ROC-AUC)이 안정적으로 0.5를 넘어서면, 그 근거를 가지고 기본값을 규칙 기반 → ML로 전환한다.

예측(predicted) vs 실측(observed)

처음엔 아직 아무도 문자를 받지 않았으니 “예측값”으로 추천한다. 그러다 실제로 문자를 보내고 열람·클릭 같은 반응 데이터가 쌓이면, 예측이 아니라 실제로 관측된 클릭률을 기준으로 분석을 바꿔 보여준다.

단계분석 기준무엇을 보나
이벤트 없음predicted_assignment모델이 계산한 평균 예측 클릭률로 임시 추천
이벤트 쌓임observed_events실제 발송·클릭 로그로 계산한 관측 클릭률
참고 · 채널별 클릭률 기준 LMS·SMS·카카오는 “열람(impression)” 집계가 불안정할 수 있어 발송 대비 클릭률을 먼저 쓰고, RCS는 열람 대비 클릭률을 쓴다. 반응 데이터는 문자 발송 업체가 보내주는 웹훅(/webhooks/message-events)으로 자동 수집한다.

11실패 · 복원

실패·복원 동작 (Fallback & Recovery)

외부 서비스(LLM·벡터DB)나 AI 출력은 언제든 실패할 수 있다. 이 시스템은 한 단계가 실패해도 전체가 멈추지 않도록, 각 단계마다 안전한 대체 경로를 두고, 대체가 일어났다는 사실을 로그·플래그로 남긴다.

단계정상 경로실패 시 복원 동작
질의 해석 (Query Plan) LLM이 자연어를 Query Plan JSON으로 변환 LLM 파서를 못 쓰면 규칙 기반 파서로 자동 대체하고 fallback_used·사유를 기록
메시지 생성 OpenAI로 3종 문구 생성 (호출당 15초 타임아웃) 검증 실패 시 실패 사유를 프롬프트에 되먹여 최대 3회 재시도. 끝내 실패하면 미검증 문구를 내보내지 않고 실패로 반환. 또한 질의가 캠페인 추천 의도(recommend_campaign)가 아니거나 참조할 캠페인 컨텍스트가 없으면 생성을 건너뛰고 0건 반환 — 이때 화면(프론트)은 임시 템플릿 문구로 폴백하므로, 캠페인 문맥이 없으면 어색한 문구가 나올 수 있다.
클릭률 예측 LightGBM ML 모델로 예측 모델 로드·예측 오류 시 규칙 기반(heuristic)으로 자동 대체하고 경고 로그 기록 (정책으로 끄면 오류 반환)
SQL 실행 검증을 통과한 SELECT만 실행 위험 SQL은 실행 전 차단, 민감 컬럼은 마스킹본으로 대체 제안 (→ 5장)
설계 원칙 “틀린 결과를 내보내느니, 안전한 기본값으로 되돌아간다.” ML 예측이 안 되면 규칙 기반으로, LLM 파서가 안 되면 규칙 기반 파서로, 메시지 검증을 못 넘기면 아예 내보내지 않는다. 모든 대체 동작은 로그와 플래그로 남겨 사후에 추적·집계할 수 있다.
질의 해석 보정 발송 채널(예: “발송 채널: RCS”)은 타겟 조건이 아니라 발송 수단이므로 SQL(오디언스·캠페인 필터)에 넣지 않고 캠페인 목적(objective)만 반영한다. 또한 “신제품 출시 소식을 알리고 싶어요”처럼 (신제품)알림·홍보 아웃리치 질의는 단순 세그먼트 조회가 아니라 캠페인 추천(recommend_campaign)으로 해석해, 타겟팅과 메시지 생성까지 자연스럽게 이어지도록 한다.

fastembed 문장을 컴퓨터가 비교할 수 있는 숫자(지문)로 바꿔주는 도구. 이 지문이 있어야 “뜻이 비슷한지”를 계산할 수 있다. 외부 유료 서비스 없이 자체 처리한다.
qdrant-client 위에서 만든 숫자 지문을 모아 둔 “의미 기반 검색 창고”에 연결하는 도구. 글자가 달라도 뜻이 비슷한 자료를 찾아준다.
networkx 고객·상품·캠페인이 서로 어떻게 연결돼 있는지(관계망)를 지도로 그리고 따라가는 도구. “이 고객과 비슷한 사람은 어떤 캠페인에 반응했나” 같은 관계를 찾는다.

lightgbm 과거 데이터를 학습해 “이 문자를 누가 클릭할까”를 예측하는 인공지능 모델 본체.
scikit-learn 그 예측 모델이 얼마나 잘 맞히는지 점수로 채점하는 도구.
pandas 데이터를 엑셀 표처럼 정리해서 다루는 도구. 학습에 쓸 자료를 불러오고 가공한다.
numpy 많은 숫자를 빠르게 계산해주는 도구. 위 표 계산과 모델 학습의 밑바탕이 된다.
joblib 학습이 끝난 모델을 파일로 저장했다가 다시 불러오는 도구. 매번 새로 학습하지 않아도 되게 해준다.
